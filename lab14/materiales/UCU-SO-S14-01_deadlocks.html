Deadlocks: cuando los procesos quedan esperando para siempre

Un deadlock ocurre cuando varios procesos quedan bloqueados porque cada uno espera un recurso que está retenido por otro. En esta presentación veremos cómo se producen, cómo identificarlos y qué estrategias puede aplicar un sistema operativo.

Conceptos principales

Analogías con la vida real

Los deadlocks son más fáciles de entender cuando los comparamos con situaciones cotidianas.

Cruce bloqueado

Cuatro autos entran a una intersección estrecha. Cada auto quiere avanzar, pero el espacio que necesita está ocupado por otro. Nadie puede retroceder ni avanzar.

Espera circular

Cena con dos utensilios

Dos personas necesitan tenedor y cuchillo para comer. Una toma el tenedor, la otra toma el cuchillo, y ambas esperan el utensilio faltante.

Retener y esperar

Impresora y escáner

Un proceso tiene la impresora y espera el escáner. Otro tiene el escáner y espera la impresora. Ninguno libera lo que ya posee.

Recursos compartidos

Las cuatro condiciones necesarias

Para que exista deadlock deben cumplirse simultáneamente las cuatro condiciones de Coffman.

Exclusión mutua

Algunos recursos solo pueden ser usados por un proceso a la vez.

Retener y esperar

Un proceso mantiene recursos asignados mientras espera otros recursos adicionales.

No apropiación

El sistema no puede quitarle por la fuerza un recurso a un proceso; debe liberarlo voluntariamente.

Espera circular

Existe una cadena circular de procesos donde cada uno espera un recurso retenido por el siguiente.

Idea clave

Si una de estas condiciones se rompe, el deadlock no puede producirse.

CondiciónCómo romperla
Exclusión mutuaUsar recursos compartibles cuando sea posible.
Retener y esperarPedir todos los recursos al inicio.
No apropiaciónPermitir que el SO recupere recursos.
Espera circularOrdenar los recursos y solicitarlos siempre en el mismo orden.

Cómo detectar un deadlock

Una forma clásica de analizar bloqueos es representar procesos y recursos mediante un grafo.

Grafo de asignación de recursos

Los procesos y recursos se representan como nodos. Una flecha de proceso a recurso indica solicitud. Una flecha de recurso a proceso indica asignación.

Ciclos en el grafo

Un ciclo puede indicar deadlock. Si cada recurso tiene una única instancia, el ciclo confirma el deadlock. Si hay múltiples instancias, se requiere un análisis adicional.

Ejemplo: P1 tiene R1 y espera R2. P2 tiene R2 y espera R1. El sistema queda bloqueado porque ningún proceso puede continuar sin que el otro libere su recurso.

Estrategias frente a deadlocks

El sistema operativo puede prevenir, evitar, detectar y recuperar, o directamente ignorar ciertos deadlocks según el contexto.

Prevención

Impide que se cumpla alguna condición necesaria del deadlock.

Reglas estrictas

Evitación

Decide si una asignación es segura antes de conceder recursos.

Estado seguro

Detección

Permite que el bloqueo ocurra y luego lo identifica.

Monitoreo

Recuperación

Libera recursos, finaliza procesos o revierte operaciones para salir del bloqueo.

Algoritmo del banquero

Es una técnica de evitación. Antes de conceder recursos, verifica si el sistema queda en un estado seguro.

Analogía bancaria

Un banco no presta todo su dinero al primer cliente. Evalúa si podrá responder a futuros retiros y préstamos. El sistema operativo hace algo parecido con los recursos.

Elementos principales

  • Disponible: recursos libres actualmente.
  • Asignado: recursos que ya tiene cada proceso.
  • Máximo: cantidad máxima que podría pedir cada proceso.
  • Necesidad: máximo menos asignado.
  • Estado seguro: existe un orden donde todos pueden terminar.
Importante: el algoritmo del banquero evita deadlocks, pero no siempre se usa en sistemas reales porque requiere conocer por adelantado la demanda máxima de recursos.

Ejercicio práctico en C++

Construir una simulación simple de procesos que compiten por recursos y pueden caer en deadlock.

Consigna

Crear un programa en C++ que simule dos procesos y dos recursos compartidos. Cada proceso debe intentar tomar los recursos en distinto orden para evidenciar una posible situación de deadlock.

Requisitos
  • Usar std::thread para representar procesos.
  • Usar std::mutex para representar recursos exclusivos.
  • Mostrar por consola cuándo un proceso toma o espera un recurso.
  • Agregar una segunda versión que evite el deadlock usando un orden fijo de adquisición.
  • Incluir una breve explicación del resultado observado.

Código base orientativo

#include <iostream>
#include <thread>
#include <mutex>
#include <chrono>

std::mutex recursoA;
std::mutex recursoB;

void proceso1() {
    std::cout << "P1 intenta tomar Recurso A\n";
    recursoA.lock();
    std::cout << "P1 tomo Recurso A\n";

    std::this_thread::sleep_for(std::chrono::milliseconds(500));

    std::cout << "P1 intenta tomar Recurso B\n";
    recursoB.lock();
    std::cout << "P1 tomo Recurso B\n";

    recursoB.unlock();
    recursoA.unlock();
}

void proceso2() {
    std::cout << "P2 intenta tomar Recurso B\n";
    recursoB.lock();
    std::cout << "P2 tomo Recurso B\n";

    std::this_thread::sleep_for(std::chrono::milliseconds(500));

    std::cout << "P2 intenta tomar Recurso A\n";
    recursoA.lock();
    std::cout << "P2 tomo Recurso A\n";

    recursoA.unlock();
    recursoB.unlock();
}

int main() {
    std::thread t1(proceso1);
    std::thread t2(proceso2);

    t1.join();
    t2.join();

    std::cout << "Fin del programa\n";
    return 0;
}
