Procesos e hilos

Proceso

Es un programa en ejecución con su propio espacio de direcciones, recursos, archivos abiertos y estado.

PCBmemoriarecursos

Hilo

Es una unidad de ejecución dentro de un proceso. Varios hilos comparten memoria, pero cada uno tiene su pila y contexto.

stackPCregistros
Idea clave: el proceso es la “casa”; los hilos son las “personas” trabajando dentro de ella, compartiendo herramientas pero realizando tareas distintas.

Sincronización y comunicación

Sincronización

Evita que varios hilos modifiquen datos compartidos al mismo tiempo de forma insegura.

Herramientas: mutex, semáforos, monitores, variables de condición.

Comunicación

Permite que procesos o hilos intercambien información.

Mecanismos: memoria compartida, pipes, sockets, colas de mensajes.

Concepto productor / consumidor

Es un patrón clásico de sincronización donde unos hilos generan datos o tareas y otros hilos las procesan.

Productor

Genera elementos: comandos, solicitudes de I/O, eventos, paquetes o registros. Los coloca en una cola compartida.

Buffer / cola

Es memoria compartida en RAM. Actúa como “zona de espera” entre quien produce y quien consume.

Consumidor

Toma elementos de la cola y los procesa: escribe en disco, imprime, envía por red o atiende un dispositivo.

Problema que resuelve

El productor puede ser más rápido que el consumidor, o el consumidor puede llegar antes de que existan datos. Por eso se necesita sincronización.

Herramientas típicas: mutex para proteger la cola, condition_variable para dormir/despertar hilos, y una condición de finalización para cerrar correctamente.

Memoria principal

RAM

Espacio rápido donde se cargan instrucciones y datos que la CPU necesita ejecutar.

Memoria virtual

Da a cada proceso la ilusión de tener una memoria propia y grande, usando paginación.

Page fault

Ocurre cuando una página necesaria no está en RAM y debe traerse desde disco.

Buffer / caché

Áreas temporales que reducen accesos lentos a disco o dispositivos.

Memoria secundaria y sistemas de archivo

Bloques

El disco se organiza en bloques. El SO traduce archivos a bloques físicos.

Directorios

Permiten ubicar archivos por rutas, nombres y permisos.

Metadatos

Tamaño, propietario, fechas, permisos y ubicación de bloques.

En C++: operaciones como ifstream, ofstream o fstream terminan pidiendo servicios al SO para leer/escribir archivos.

Gestión de I/O

Colas

Ordenan solicitudes pendientes de dispositivos.

Temporizadores

Controlan tiempos de espera, quantums y timeouts.

Interrupciones

Permiten reaccionar a eventos sin consultar constantemente.

Mini simulación productor/consumidor en C++

// Idea base: productor-consumidor con hilos, cola, mutex y archivo.
#include <iostream>
#include <fstream>
#include <thread>
#include <mutex>
#include <condition_variable>
#include <queue>
#include <chrono>
using namespace std;

queue<string> eventos;
mutex mtx;
condition_variable cv;
bool finalizar = false;

void productorEntrada() {
    for (int i = 1; i <= 5; i++) {
        this_thread::sleep_for(chrono::milliseconds(700)); // temporizador
        lock_guard<mutex> lock(mtx);
        eventos.push("Comando recibido " + to_string(i)); // cola en RAM
        cv.notify_one();
    }
    lock_guard<mutex> lock(mtx);
    finalizar = true;
    cv.notify_one();
}

void consumidorLog() {
    ofstream log("eventos.log"); // sistema de archivo
    while (true) {
        unique_lock<mutex> lock(mtx);
        cv.wait(lock, [] { return !eventos.empty() || finalizar; });
        if (eventos.empty() && finalizar) break;
        string e = eventos.front(); eventos.pop();
        lock.unlock();
        log << e << endl; // I/O a disco
        cout << "Procesado: " << e << endl;
    }
}

int main() {
    thread t1(productorEntrada);
    thread t2(consumidorLog);
    t1.join();
    t2.join();
}

Ejercicio práctico integrador

Simulador de atención de solicitudes de I/O

Implementar en C++ un programa que simule un sistema operativo atendiendo solicitudes de lectura/escritura realizadas por varios procesos.

Requisitos

  • Crear al menos 3 hilos productores, representando procesos.
  • Cada proceso genera solicitudes de I/O y las coloca en una cola compartida.
  • Un hilo consumidor representa el driver/dispositivo.
  • Usar mutex y condition_variable para sincronizar.
  • Guardar en un archivo log cada solicitud atendida.
  • Simular temporizadores con sleep_for.

Integración esperada

  • Procesos: productores concurrentes.
  • Planificación: observar intercalado de ejecución.
  • Memoria: cola compartida en RAM.
  • Archivo: persistencia en log.
  • I/O: cola, espera, atención y finalización.

Guía de resolución

1. Definir la estructura SolicitudIO

Debe tener idProceso, tipoOperacion, archivoDestino y tiempoEstimado.

2. Crear la cola compartida

Usar queue<SolicitudIO>, mutex y condition_variable.

3. Programar productores

Cada hilo genera solicitudes con pausas aleatorias.

4. Programar consumidor

Extrae solicitudes, simula el tiempo de I/O y escribe en el log.

5. Verificar salida

Comparar consola y archivo para explicar planificación, sincronización y almacenamiento.

¿Qué pasaría si quitamos el mutex?

Cierre

Un Sistema Operativo coordina ejecución, memoria, persistencia y dispositivos. Cuando un programa aparentemente simple lee, calcula y guarda datos, el SO ya está planificando hilos, administrando memoria, sincronizando recursos y gestionando I/O.
